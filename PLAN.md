**Kernprinzipien:**
1. **Repo = Lib** вҖ” `Knaackee/agentsmd` enthГӨlt alle Agents/Skills/Workflows unter `/lib/`
2. **Self-contained Exe** вҖ” .NET 10, keine Dependencies, GitHub Actions baut fГјr Win/Mac/Linux
3. **Auto-Updater** вҖ” CLI prГјft bei Start ob neue Version im Repo vorhanden ist
4. **Front Matter = Metadata** вҖ” jede Datei trГӨgt Name, Version, Beschreibung, Deps
5. **Dependency Resolution** вҖ” `agentsmd workflow install sdd-tdd` installiert automatisch benГ¶tigte Agents + Skills
6. **Ein Ordner = alles** вҖ” `.agentsmd/` ist Source of Truth + CLI-State. Minimaler Footprint
7. **Sync = Propagation** вҖ” `agentsmd sync` generiert Tool-Wrappers + kopiert Skills wohin Tools sie erwarten
8. **Skills = Ordner** вҖ” ein Skill kann Scripts (`.sh`, `.ps1`, `.py`) enthalten, nicht nur Markdown

**Konventionen:**

### Lib-Index (`index.json`)

Eine GitHub Action generiert bei jedem Push auf `/lib/` einen Index. Die CLI fetcht **eine Datei** statt dutzende API-Calls.

```json

  "generated": "2026-03-21T12:00:00Z",
  "artifacts": [
    {
      "name": "test-writer",
      "type": "agent",
      "version": "1.2.0",
      "description": "Schreibt Tests aus Spezifikationen",
      "path": "agents/test-writer.md",
      "deps": [],
      "tags": ["testing", "tdd", "sdd"]
    },
    {
      "name": "feature",
      "type": "skill",
      "version": "1.0.0",
      "description": "Feature-Entwicklung mit SDD",
      "path": "skills/feature/",
      "deps": [],
      "tags": ["feature", "sdd"]
    },
    {
      "name": "sdd-tdd",
      "type": "workflow",
      "version": "2.0.0",
      "description": "Spec-Driven Development mit TDD Pipeline",
      "path": "workflows/sdd-tdd.md",
      "deps": [
        "agent:test-writer@>=1.0",
```

**Generierung:** GitHub Action bei Push auf `lib/**` вҶ’ liest alle Front Matter вҶ’ schreibt `lib/index.json`
**CLI-Nutzung:** `agentsmd search` holt nur `index.json` (1 Request), durchsucht lokal

### Front Matter Spec

Jede Lib-Datei hat YAML Front Matter:

```yaml
---
name: test-writer
type: agent                        # agent | skill | workflow
version: 1.2.0
description: Schreibt Tests aus Spezifikationen
deps:                               # Dependencies (optional)
  - agent:implementer@>=1.0
  - skill:feature@>=1.0
tags:                               # FГјr Search
  - testing
  - tdd
  - sdd
---

# Test-Writer Agent
...eigentlicher Inhalt...
```

**Dependency-Syntax:**
- `agent:name@version` вҖ” benГ¶tigt Agent
- `skill:name@version` вҖ” benГ¶tigt Skill
- `workflow:name@version` вҖ” benГ¶tigt Workflow
- Version: `>=1.0`, `^2.0`, `1.2.0` (exakt), `latest`

### Workflows

Ein Workflow ist eine Orchestrierungs-Datei: beschreibt **welche Agents in welcher Reihenfolge**
mit welchen Ein-/Ausgaben laufen. Dependencies im Front Matter sorgen dafГјr, dass
`agentsmd workflow install` automatisch alle benГ¶tigten Agents + Skills mitinstalliert.

**Beispiel: SDD+TDD Workflow**
```yaml
---
name: sdd-tdd
type: workflow
version: 2.0.0
description: Spec-Driven Development mit TDD Pipeline
deps:
  - agent:test-writer@>=1.0
  - agent:implementer@>=1.0
  - agent:reviewer-spec@>=1.0
  - agent:reviewer-quality@>=1.0
  - agent:docs-writer@>=1.0
  - skill:feature@>=1.0
  - skill:task-management@>=1.0
tags:
  - development
  - tdd
  - sdd
---

# SDD+TDD Workflow

## Pipeline
FГјr jede Aufgabe in TASKS.md:

1. **RED** вҶ’ `test-writer` вҶ’ Failing Tests schreiben
2. **GREEN** вҶ’ `implementer` вҶ’ Minimaler Code bis Tests grГјn
3. **SPEC-REVIEW** вҶ’ `reviewer-spec` вҶ’ Alle Kriterien erfГјllt?
4. **QUALITY-REVIEW** вҶ’ `reviewer-quality` вҶ’ Code-QualitГӨt + Security
5. **DOCS** вҶ’ `docs-writer` вҶ’ Dokumentation aktualisieren
6. **COMMIT** вҶ’ Alles grГјn вҶ’ Commit

## Execution Modes
- **autonomous**: LГӨuft alle Tasks durch ohne Pause
- **guided**: Wartet nach jedem Task auf "next task"
```

**Beispiel: Nicht-Programmierung**
```yaml
---
name: content-review
type: workflow
version: 1.0.0
description: Content-Erstellung mit Review-Pipeline
deps:
  - agent:content-writer@>=1.0
  - agent:editor@>=1.0
  - agent:fact-checker@>=1.0
  - skill:task-management@>=1.0
tags:
  - content
  - writing
---

# Content Review Workflow

1. **DRAFT** вҶ’ `content-writer` вҶ’ Entwurf erstellen
2. **FACT-CHECK** вҶ’ `fact-checker` вҶ’ Fakten verifizieren
3. **EDIT** вҶ’ `editor` вҶ’ Sprache + Stil Гјberarbeiten
4. **PUBLISH** вҶ’ Final
```

вҶ’ Workflows sind **nicht an Programmierung gebunden**. Jeder Anwendungsfall mit einer Pipeline aus Agents kann ein Workflow sein.

**Regeln:**
- Projekt initialisiert? вҶ’ `.agentsmd/` existiert
- Was installiert? вҶ’ `.agentsmd/installed.json`
- Welche Tools aktiv? вҶ’ Wrapper-Ordner vorhanden (`.github/agents/`, `.opencode/agents/`, `.claude/agents/`)
- Skills fГјr Claude Code? вҶ’ Copy von `.agentsmd/skills/` вҶ’ `.claude/skills/`
- Workflows? вҶ’ direkt aus `.agentsmd/workflows/` (kein Copy nГ¶tig)
- Tasks? вҶ’ direkt aus `.agentsmd/tasks/` (backlog + in-progress)
- Lib-URL? вҶ’ hardcoded: `github.com/Knaackee/agentsmd` (das Repo selbst, `/lib/` Ordner)

## Architektur-Entscheidungen

### Tech Stack

| Komponente | Wahl | BegrГјndung |
|-----------|------|------------|
| Sprache | **.NET 10 / C#** | Self-contained Exe, keine Runtime nГ¶tig, cross-platform |
| CLI Framework | **System.CommandLine** | Microsoft-eigenes CLI Framework, robust |
